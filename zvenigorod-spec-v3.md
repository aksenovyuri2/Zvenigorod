# Мэр Звенигорода — v3: Прорыв

> Дополнение к v1 + v2. Всё что ниже — поверх существующих механик. Приоритет — v3.

---

## ФИЛОСОФИЯ V3

V1–V2 — это настолка: метрики, решения, кубик. Хорошая, но предсказуемая.

V3 превращает игру в **живой город**. Каждая партия — уникальная история. Жители — не числа, а люди с именами. Решения — не «+12 к инфраструктуре», а «мост, который строится 3 квартала, подрядчик ворует, жители пишут жалобы, а газета публикует расследование». Мэр — не оператор ползунков, а политик, который лавирует между фракциями, держит обещания и решает, брать ли взятку.

Три столпа v3:
1. **AI-нарратор** — Claude API генерирует уникальные тексты: газету, письма жителей, речи на думе, итоги квартала
2. **Живой город** — NPC-жители, бизнесы, фракции, СМИ как самостоятельные агенты
3. **Глубокие системы** — экономика, политика, кризисы, дипломатия — всё взаимосвязано

## 1. AI-НАРРАТОР (Claude API)

### 1.1 Концепция

Каждый текст в игре, кроме UI-лейблов, генерируется Claude API. Это делает каждую партию уникальной и создаёт ощущение живого мира. AI получает полный контекст (state, историю решений, персонажей) и генерирует текст в стиле и тоне, подходящем контексту.

### 1.2 Точки генерации

| Триггер | Что генерируется | Макс. длина | Температура |
|---|---|---|---|
| Начало хода | Вступительный абзац «состояние города» | 100 слов | 0.8 |
| Событие | Описание события (вместо шаблона) | 150 слов | 0.9 |
| Итоги квартала | Нарративное резюме квартала | 200 слов | 0.7 |
| Письмо жителя | Письмо от NPC-жителя мэру | 100 слов | 0.9 |
| Газетная статья | Статья в «Звенигородских вестях» | 150 слов | 0.8 |
| Реплика советника | Комментарий к решению | 50 слов | 0.8 |
| Речь на думе | Выступление фракции за/против решения | 80 слов | 0.8 |
| Выборы | Предвыборные дебаты и речь оппонента | 150 слов | 0.9 |
| Финал игры | Эпилог: «Что стало с городом через 10 лет» | 300 слов | 0.9 |

### 1.3 Контекст для AI

Каждый запрос к API включает сжатый game state:

```typescript
interface NarratorContext {
  turn: number;
  season: string;
  year: number;
  metrics: Metrics;
  metricDeltas: Metrics;             // изменения за последний ход
  approval: number;
  population: number;
  budget: number;
  debt: number;
  rank: number;
  nearestCities: string[];           // 2 города выше и ниже в рейтинге
  recentDecisions: string[];         // решения последних 3 ходов
  recentEvents: string[];            // события последних 3 ходов
  activeProjects: string[];          // незавершённые проекты
  criticalMetrics: string[];         // метрики < 25
  boostingMetrics: string[];         // метрики > 75
  npcContext?: NPCContext;           // если генерим письмо конкретного NPC
  advisorContext?: AdvisorContext;   // если генерим реплику советника
  electionContext?: ElectionContext; // если генерим предвыборный контент
  cityVisualState: string;           // краткое описание визуального состояния карты
  previousNarratives: string[];      // последние 2 нарратива (для continuity)
}
```

### 1.5 Fallback при отсутствии API / ошибках

Если API недоступен — используются шаблонные тексты из v2. В UI — незаметный переключатель. Игра полностью играбельна без AI. AI = premium layer поверх.

### 1.6 Кэширование и оптимизация

- Все запросы к API — async, не блокируют UI
- Пока текст генерируется — показывать typing-анимацию (три точки)
- Кэшировать нарративы в state (не перегенерировать при перезагрузке хода)
- Максимум 3–4 API-вызова за ход (batch где возможно)
- model: `claude-sonnet-4-20250514`, max_tokens: 300

## 2. ЖИВЫЕ ЖИТЕЛИ (NPC-СИСТЕМА)

### 2.1 Концепция

Вместо абстрактных «групп населения» — 80 конкретных NPC, которые живут в городе, реагируют на решения, пишут письма, уезжают и приезжают. Группы остаются как аналитическая надстройка, но эмоциональное ядро — конкретные люди.

### 2.2 Структура NPC

```typescript
interface NPC {
  id: string;
  name: string;                  // "Галина Фёдоровна Крылова"
  shortName: string;             // "Галина Фёдоровна"
  age: number;
  group: GroupKey;               // elderly | freelancers | families | youth | business
  avatar: string;                // сгенерированный emoji-аватар или pixel-art
  district: string;              // "Верхний Посад", "Саввинская слобода"...
  occupation: string;            // "пенсионерка", "веб-дизайнер", "владелец кафе"
  personality: PersonalityType;  // optimist | pessimist | activist | quiet | pragmatist
  needs: MetricKey[];            // 2–3 ключевые потребности (healthcare, safety...)
  satisfaction: number;          // 0–100, индивидуальная
  loyalty: number;               // 0–100, лояльность к мэру
  residenceTurn: number;         // с какого хода живёт в городе (0 = с начала)
  status: "resident" | "moved_out" | "moved_in";
  memory: NPCMemory[];          // помнит решения мэра, лично затронувшие
  lastLetterTurn: number;        // когда последний раз писал письмо
}

interface NPCMemory {
  turn: number;
  event: string;                 // "Построили детсад рядом с моим домом"
  sentiment: "positive" | "negative";
}

type PersonalityType = "optimist" | "pessimist" | "activist" | "quiet" | "pragmatist";
```

### 2.3 Генерация NPC

При старте игры генерируется 60 NPC (стартовые жители). Распределение:
- elderly: 15
- freelancers: 8
- families: 15
- youth: 12
- business: 10

Каждый — уникальная комбинация имени, района, профессии, характера. Имена из предзаготовленного пула (200+ русских имён/фамилий).

### 2.4 Динамика NPC

**Каждый ход:**
- Пересчитать satisfaction каждого NPC на основе его needs и текущих метрик
- Если satisfaction < 20 в течение 3 ходов → NPC **уезжает** (статус `moved_out`)
  - Показать уведомление: *«Семья Петровых уехала в Казань. Причина: плохая экология.»*
- Если город привлекателен → новые NPC **приезжают** (статус `moved_in`)
  - Показать: *«Дизайнер Антон Лебедев (28) переехал из Москвы. Привлекло: коворкинг и природа.»*

**Письма мэру:**
- Каждый ход 1–2 NPC пишут письмо (выбираются по экстремальности satisfaction — самые довольные и самые недовольные)
- Текст письма генерируется AI с учётом personality, needs, memory NPC
- Письма показываются в специальном «Почтовом ящике мэра» 📬

**Память NPC:**
- NPC помнят решения, которые напрямую затрагивают их needs
- Это влияет на loyalty и тон писем
- Пример: Галина Фёдоровна помнит, что мэр модернизировал поликлинику → loyalty +20, пишет благодарность

## 3. ПОЛИТИЧЕСКАЯ СИСТЕМА

### 3.1 Городская дума (7 депутатов)

Дума — не декорация, а реальный gameplay-элемент. Некоторые решения (дорогие, спорные) требуют одобрения думы. Мэр должен убеждать депутатов.

```typescript
interface DeputyNPC {
  id: string;
  name: string;
  faction: FactionKey;
  avatar: string;
  loyalty: number;           // 0–100 к мэру
  corruptibility: number;    // 0–100 насколько подвержен давлению
  priorities: MetricKey[];   // за что голосует охотнее
  voteHistory: VoteRecord[];
}

type FactionKey = "progress" | "tradition" | "business" | "social" | "independent";
```

### 3.2 Фракции

| Фракция | Кол-во | Приоритеты | Описание |
|---|---|---|---|
| «Прогресс» | 2 | digital, economy, culture | Молодые, хотят инноваций и роста |
| «Традиция» | 2 | culture, ecology, safety | Консерваторы, защищают историческое наследие |
| «Бизнес-клуб» | 1 | economy, infrastructure | Предприниматели, хотят выгоду |
| «Социальная справедливость» | 1 | healthcare, education, safety | За пенсионеров и детей |
| Независимый | 1 | random | Непредсказуем, голосует по ситуации |

### 3.3 Механика голосования

**Какие решения требуют голосования думы:**
- Стоимость > 120 млн
- Решения с tradeoff (негативные эффекты)
- Решения с `once: true` и стоимостью > 80
- Изменения налоговой политики
- Все решения из событий-с-выбором, где бюджетный эффект > 50

**Процесс:**

```
1. Мэр выбирает решение, требующее голосования
2. Экран думы: 7 депутатов, их позиции (за/против/воздержался)
3. Позиция зависит от:
   - Совпадение приоритетов фракции с эффектами решения
   - Loyalty к мэру
   - Текущее состояние города
4. Мэр может ПЕРЕД голосованием:
   - Произнести речь (AI-generated, +5–15 к loyalty склоняющихся)
   - Предложить компромисс (уменьшить масштаб решения: -30% cost и -30% effects)
   - Оказать давление на одного депутата (+20 loyalty ему, -10 loyalty остальным)
5. Голосование: 4 из 7 = принято, иначе заблокировано
6. Заблокированное решение нельзя предлагать ещё 2 хода
```

### 3.5 Коррупция и моральные дилеммы

Раз в 3–5 ходов возникают «серые» ситуации:

**Взятка от застройщика:**
> Представитель строительной компании «МосДевелоп» предлагает вам 150 млн в городской бюджет... и 50 млн «на развитие». Взамен — разрешение на высотную застройку в историческом центре.
>
> [Принять всё] бюджет +200, infrastructure +5, culture -8, approval -5 (если узнают)
> [Только в бюджет] бюджет +150, infrastructure +3
> [Отказать и предать огласке] culture +5, approval +10, economy -3

**Риск разоблачения:** если мэр берёт «серые» деньги — есть 30% шанс через 2–4 хода: «Журналист раскопал коррупционную схему мэра» → approval -25, возможный импичмент.

**Импичмент:** если approval падает ниже 15 — дума инициирует голосование об отставке. 5 из 7 голосов = game over.

### 3.6 Петиции жителей

Новая механика: жители собирают подписи за/против чего-то. Петиция = мини-событие:

```typescript
interface Petition {
  title: string;           // "За сохранение леса в Дютьково"
  initiator: string;       // NPC id
```

## 4. КРИЗИСНАЯ СИСТЕМА

### 4.1 Концепция

Кризис — не одноходовое событие, а **многоходовая цепочка** (3–6 ходов) с нарастающими последствиями и emergency-решениями. За игру происходит 2–4 кризиса.

### 4.2 Структура кризиса

```typescript
interface Crisis {
  id: string;
  name: string;
  icon: string;
  phases: CrisisPhase[];           // фазы кризиса (3–6)
  triggerCondition: string;        // когда триггерится
  minTurn: number;
  maxTurn: number;
}

interface CrisisPhase {
  turn: number;                    // на каком ходу после начала
  description: string;            // что происходит (или AI-шаблон)
  autoEffects: Partial<Effects>;  // автоматические последствия
  emergencyActions: EmergencyAction[];  // что может сделать мэр
}

interface EmergencyAction {
  name: string;
  cost: number;
  effects: Partial<Effects>;
  description: string;
}
```

## 6. ЭКОНОМИЧЕСКАЯ ГЛУБИНА

### 6.1 Детализированная экономика

Вместо одной метрики `economy` — развёрнутая экономическая панель:

```typescript
interface CityEconomy {
  gdp: number;                    // ВГП (валовый городской продукт) в млн ₽
  unemployment: number;           // % безработицы (0–30)
  avgSalary: number;              // средняя зарплата ₽/мес
  housingPrice: number;           // стоимость м² жилья ₽
  costOfLiving: number;           // индекс стоимости жизни (50–200, Москва = 100)
  businessCount: number;          // количество бизнесов
  taxRate: number;                // муниципальная ставка (5–15%)
  inflation: number;              // инфляция %
}
```

**Формулы:**

```
gdp = businessCount × avgRevenue × (1 - unemployment/100) × population/25000
avgSalary = baseGDP_perCapita × (1 + economy/200) × (1 - unemployment/50)
housingPrice = basePriceM2 × (1 + population/50000) × (1 + attractiveness/200)
costOfLiving = 60 + housingPrice/1500 + inflation × 2
unemployment = max(0, 15 - economy × 0.12 - businessCount × 0.05)
```

### 6.2 Налоговая политика

Мэр может менять налоговую ставку (раз в 4 хода):

| Ставка | Эффект на revenue | Эффект на economy | Эффект на бизнесы |
|---|---|---|---|
| 5% (низкая) | revenue × 0.7 | economy +4 | бизнесы приходят |
| 8% (ниже средней) | revenue × 0.85 | economy +2 | нейтрально |
| 10% (стандарт) | revenue × 1.0 | 0 | нейтрально |
| 12% (выше средней) | revenue × 1.1 | economy -2 | бизнесы уходят медленно |
| 15% (высокая) | revenue × 1.2 | economy -5 | бизнесы уходят быстро |

### 6.3 Рынок недвижимости

Цена жилья влияет на:
- Привлекательность для freelancers и youth (дешевле = лучше)
- Привлекательность для business и families (дороже = «хороший район»)
- Объём налоговых поступлений

Если цена растёт слишком быстро → событие «Жители протестуют против дороговизны жилья»
Если падает → событие «Инвесторы теряют интерес»

### 6.4 Бизнесы как сущности

5–15 NPC-бизнесов, аналогично жителям:

```typescript
interface BusinessNPC {
  id: string;
  name: string;               // "Пекарня «У Марины»"
  type: BusinessType;          // cafe | shop | it | farm | service | factory
  owner: string;               // NPC id владельца
  revenue: number;             // квартальная выручка
  employees: number;
  satisfaction: number;        // 0–100
  district: string;
  openedTurn: number;
}
```

Бизнесы:
- Открываются при economy > 40 и economy растёт
- Закрываются при economy < 30 или если их тип метрика < 25
- Каждый бизнес = +X к revenue города и +Y к economy
- При закрытии — показывается уведомление: *«Кафе «Берёза» на Московской закрылось. Мария Белова: «Аренда сожрала весь доход»»*

## 10. ПРОЕКТЫ КАК СУЩНОСТИ

### 10.1 Концепция

Дорогие решения (> 80 млн) — не мгновенные. Они становятся **проектами** с этапами строительства, рисками и прогрессом.

### 10.2 Структура проекта

```typescript
interface Project {
  decisionId: string;
  name: string;
  totalCost: number;
  paidCost: number;              // сколько уже потрачено
  costPerTurn: number;           // стоимость за ход строительства
  totalTurns: number;            // сколько ходов строится (2–5)
  currentTurn: number;           // текущий ход строительства
  status: "building" | "completed" | "stalled" | "cancelled";
  riskPerTurn: number;           // вероятность проблемы (5–20%)
  partialEffects: Partial<Effects>;   // частичные эффекты во время строительства
  fullEffects: Partial<Effects>;      // полные эффекты по завершении
  contractor: Contractor;
}

interface Contractor {
  name: string;                  // "ООО «СтройМастер»"
  reliability: number;           // 0.5–1.0
  costModifier: number;          // 0.8–1.3 (дешёвый но ненадёжный vs дорогой но надёжный)
}
```

### 10.3 Выбор подрядчика

При запуске проекта — игрок выбирает из 2–3 подрядчиков:

```

│     Цена: 150 млн (стандарт)                            │
```

### 10.4 Риски проекта

Каждый ход строительства — бросок на риск:

```
risk_roll = random(0, 100)
if risk_roll < riskPerTurn × (1 / contractor.reliability):
  // проблема!
```

Проблемы (рандом):
- **Задержка:** +1 ход к сроку строительства
- **Перерасход:** +20% к оставшейся стоимости
- **Скандал:** подрядчик ворует → approval -3, газета пишет статью
- **Брак:** частичные эффекты снижены на 50%

Мэр может реагировать:
- «Сменить подрядчика» — стоимость +30%, но надёжность повышается
- «Провести проверку» — 20 млн, выявляет проблему + approval +2
- «Закрыть глаза» — бесплатно, но если потом вскроется — approval -8

### 10.5 Визуализация на карте

Незавершённые проекты на карте — в виде стройки (леса, кран). Прогресс-бар на тайле. По завершении — анимация «открытие»: лента, люди, фейерверк.

## 14. КЛЮЧЕВЫЕ ПРИНЦИПЫ ДИЗАЙНА V3

1. **Каждое решение — дилемма.** Не бывает «правильного» ответа. Всегда есть tradeoff.
2. **Последствия приходят с задержкой.** Решения аукнутся через 2–5 ходов. Игрок учится думать на перспективу.
3. **Люди > числа.** Семья Петровых уехала — это больнее, чем «население -200».
4. **AI делает каждую партию уникальной.** Два игрока с одинаковыми решениями получат разный нарратив.
5. **Кризисы — не наказание, а возможность.** Хороший мэр превращает кризис в трамплин.
6. **Политика — грязная.** Дума блокирует, оппозиция врёт, подрядчики воруют. Это реализм, и это интересно.
7. **Город живой.** Карта меняется, люди приходят и уходят, бизнесы открываются и закрываются. Ты чувствуешь, что это не spreadsheet, а город.
